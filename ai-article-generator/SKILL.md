# AI Article Generator

## 总则

用这个 skill 把“参考文章 + 主题”变成可发布文章。用户只负责选参考文章、选主题、选标题/大纲；你负责拆解、提炼、写作、校审和生成可复制的本地产物。

默认中文。默认读者是有本科受教育背景的成年人，但对 AI、互联网、产品、商业等专业概念不熟。少讲术语，首次出现概念时用一句话解释，并给简单例子。

## 输入

必须有两个输入：

- `参考文章`：链接、文件、截图或粘贴正文均可。
- `主题`：用户想写的新文章主题。

可选输入：

- `平台`：默认公众号/长文；如果用户指定人人都是产品经理、知乎、小红书、飞书等，按平台调整。
- `素材`：用户自己的笔记、观点、案例、产品体验、访谈记录。
- `字数`：默认先输出标题和大纲；正式成文默认 3000-5000 字。
- `身份`：默认“资深互联网产品经理/行业观察者”；如果主题不是互联网，换成对应领域的可信身份。

缺关键输入时只问一个问题。链接打不开时，不要猜正文，让用户粘贴文章内容或截图。

## 工作流

### 1. 拆框架

先分析参考文章，而不是直接仿写。提取：

- 标题：标题句式、卖点、冲突点、目标读者。
- 大纲：章节顺序，每一节解决什么问题。
- 内容：核心论点、案例、数据、观点、结论。
- 形式：开头方式、小标题风格、段落节奏、是否有清单/方法论。
- 写作风格：专业度、口语度、情绪强度、是否适合平台。
- 受欢迎原因：为什么容易被推荐、收藏、转发或评论。

如果用户只要求“总结参考文章”，先输出这一步。

### 2. 去提炼

把参考文章变成可复用写作框架。不要照搬原文表达，只保留方法：

- 可复用标题公式。
- 4-6 个章节的大纲骨架。
- 每一节的写作任务。
- 适合替换成用户主题的案例位置。
- 开头钩子：比喻、反问、警示句、具体场景，选一种即可。
- 结尾方法论：读者看完能知道下一步怎么做。

内部可使用 `references/article-brief-template.md` 组织信息。

### 3. 给用户选标题和大纲

正式写正文前，先给用户 3-5 个标题和 2-3 套大纲，让用户选。除非用户明确说“直接写”，否则不要跳过这个选择环节。

标题要具体，不要空泛。大纲要能看出文章逻辑，不要只有概念词。

### 4. 开始写

用户选定标题和大纲后再成文。写作要求：

- 开篇有冲击力：用具体场景、反问、对比、警示句或比喻切入。
- 结构完整：有章节、小标题、论点、例子和过渡。
- 用人话：少用“赋能、闭环、生态、抓手、壁垒、底层逻辑”等套话；换成具体场景、真实动作和清楚判断。
- 有方法论：最后给读者可执行的方法、步骤或判断标准。
- 不编造：涉及数据、公司、政策、时间、产品结论，必须来自可信来源；不确定就说明。
- 不抄袭：学习参考文章的结构和写法，不复制句子和独特表达。

### 5. 校审

交付前检查：

- 是否围绕用户主题，而不是复述参考文章。
- 标题、大纲、正文是否一致。
- 是否有 AI 味、套话、空话。
- 专业概念是否对新手友好。
- 数据、公司、政策、时间、产品结论是否有依据。
- 是否有明显事实错误、逻辑跳跃、过度断言。

如果用户要求“校审”，只做校审和修改建议；如果用户要求“直接改”，输出改后版本。

### 6. 正文交付格式

用户确认标题和大纲后，正式正文不要只在聊天里输出一大片文字。必须同时生成可审核、可复制的本地产物：

- `outputs/{标题}_待审核.txt`：正文源文件，使用适合人工审核的 Markdown 风格结构。
- `outputs/{标题}_待审核.md`：同内容备份，便于 Markdown 预览或后续处理。
- `outputs/article_artifact_preview.html`：仿 Claude Artifact 的本地富文本预览页。
- `outputs/article_visual_paste.html`：同一预览页的兼容别名，旧链接仍可打开。

正文源文件格式要求：

- 第一行用 `# 标题`。
- 导语用 `>` 引用块。
- 章节标题用 `## 一、...` 这种层级。
- 章节之间可用 `---` 分隔。
- 重点句用 `**加粗**`，不要依赖字体颜色。
- 不要输出平台无法识别的复杂 HTML、CSS 或花哨 Markdown。

富文本预览页要求：

- 页面顶部必须有清楚的文字按钮，至少包含 `复制正文` 和 `复制全文`，不要使用无文字图标按钮。
- `复制正文` 不含大标题，适合粘贴到“人人都是产品经理”等平台的正文区。
- `复制全文` 包含大标题，作为备用。
- 点击按钮时同时写入 `text/html` 和 `text/plain`，让网页编辑器优先粘贴已渲染富文本，而不是 Markdown 源码。
- 页面必须使用 UTF-8，不能出现中文乱码或问号替代文字。
- 页面必须提供“备用手动复制区”：如果浏览器拦截按钮复制，用户可以点击该区域后 `Ctrl+A`、`Ctrl+C` 复制已渲染正文。
- 预览页样式要接近干净的文章阅读页：标题、引用、分割线、小标题、段落和列表都要可视化。

生成预览页时，优先使用 bundled script：

```powershell
python .codex\skills\ai-article-generator\scripts\generate_article_artifact.py --source outputs\{标题}_待审核.txt --out-dir outputs
```

如果要避免中文路径在 Windows 命令里乱码，可以先进入项目根目录，并把 `--source` 写成相对路径；脚本会按 UTF-8 读取正文。

交付前验证：

- 打开 `http://127.0.0.1:8765/article_artifact_preview.html`，如果本地服务未启动，先在 `outputs` 目录启动本地 HTTP 服务。
- 刷新页面后确认按钮文字不是空白、不是乱码。
- 尽量实际点击 `复制正文`，确认剪贴板不是空内容；能检查时确认剪贴板含 `text/html` 和 `text/plain`。
- 最终回复给用户时提供本地预览链接和文件路径，不要把整篇正文重新塞满聊天窗口。

### 7. 正文交付后的固定追问

每次生成完整正文并交付本地产物后，必须在结尾主动提示用户是否继续做两件事：

1. `文章复检`：从标题、大纲、逻辑、事实依据、AI 味、表达自然度、读者可执行方法论这几个角度复查。
2. `改写成另一平台版本`：例如公众号、知乎、小红书、人人都是产品经理、飞书内部分享。

不要默认直接开始下一步，除非用户明确说“继续做复检”“改写平台版本”或“两个都做”。

提示用户时，给出简短选择：

```text
正文已完成。接下来你要不要我继续做：
1. 文章复检
2. 改写成另一平台版本
3. 两个都做
```

默认不要生成配图、封面或图片提示词，除非用户明确提出。

## 默认输出顺序

第一次生成时：

1. 参考文章拆解
2. 可选标题
3. 可选大纲
4. 等用户选择

用户选择后：

1. 生成完整文章到 `outputs/{标题}_待审核.txt` 和 `.md`
2. 生成 `article_artifact_preview.html` 富文本预览页
3. 启动或复用本地 HTTP 服务，并给用户预览链接
4. 简短说明已做的基础校审
5. 固定询问是否继续做文章复检、改写成另一平台版本，或两个都做

## 常用提示词骨架

拆框架：

```text
这是一篇高热参考文章：{reference}
请分析它的标题、大纲、内容、形式和写作风格，并说明它好在哪里，为什么容易获得推荐。
```

结合新主题：

```text
现在我需要写一篇主题为“{topic}”的文章。请根据参考文章的结构、大纲、标题、内容、形式和写作风格，结合我的素材“{notes}”，先输出标题和大纲供我选择。
```

正式成文：

```text
请按我选定的标题和大纲写完整文章。要求结构完整、有章节和小标题，开篇有冲击力，表达像人话，最后给到读者可执行的方法论。涉及数据、公司、政策、时间、产品结论时必须有可信来源，不能凭空捏造。
```
